How-To Guide

Reimage The Cluster

Remove the OpenShift node VMs and recreate them from the current appliance base image and config ISO.

Start Here GitHub Repo

Use this guide when the cluster should be destroyed and redeployed for a customer demonstration. It does not reclaim foundry or virtualization host setup.

Reimage With The Existing Base Image

This is the normal workshop reset path. It refreshes installer state and node overlays, then reuses the existing appliance-base.qcow2.

./scripts/10-prepare-appliance-assets.sh
./scripts/12-create-cluster-config-image.sh

./scripts/14-destroy-ocp-vms.sh
./scripts/13-create-ocp-vms.sh
./scripts/15-watch-ocp-install.sh
./scripts/16-verify-ocp-cluster.sh

Reimage After Mirrored Content Changes

If operators, additional images, or registry auth changed, rebuild appliance.raw and force script 13 to refresh the virtualization host base image.

./scripts/10-prepare-appliance-assets.sh
./scripts/11-build-appliance-image.sh
./scripts/12-create-cluster-config-image.sh

./scripts/14-destroy-ocp-vms.sh
APPLIANCE_REFRESH_BASE_IMAGE=true ./scripts/13-create-ocp-vms.sh
./scripts/15-watch-ocp-install.sh
./scripts/16-verify-ocp-cluster.sh

Guardrails

  • Script 14 removes only the OpenShift VM domains and overlay disks.
  • Do not use this path to recreate foundry. Foundry has a separate rebuild procedure in the Markdown runbook.
  • Script 12 starts a new installer state, including the foundry-local kubeconfig used by scripts 15 and 16.
The reimage path is meant for fast, repeatable workshop resets.
